01 项目概览

项目本质上是一个桌面化的 AI Coding Agent 控制台。它既不是纯前端，也不是传统 B/S 后端，而是由 Tauri 桌面壳启动本地 Next.js 服务端，再通过浏览器式 WebView 与本地 Agent 运行时通信。

定位

本地桌面 AI Coding Agent 客户端，重点是会话管理、工具控制、多代理编排、扩展接入。

运行方式

Tauri 在开发环境加载 localhost:30141，生产环境启动内嵌 Node + Next standalone 服务。

数据核心

会话以 jsonl 文件落盘，位置为 ~/.deerhux/agent/sessions。

02 总体架构

系统的关键边界是：浏览器/WebView 只读读取会话；发送消息必须进入 Next.js API，再由进程内的 AgentSessionWrapper 驱动 pi-coding-agent。

核心设计判断：会话浏览与会话执行被刻意拆开。前者读取 jsonl 文件，不创建 AgentSession；后者通过 rpc-manager 创建或复用进程内 wrapper。

04 Agent 核心引擎：rpc-manager.ts

rpc-manager.ts 是整个项目最关键的文件。它负责 AgentSessionWrapper、命令分发、事件转发、工具启停、角色/模式/系统提示词注入、MCP 动态装载、子 Agent 工具注册、空闲销毁等。

命令分发清单

关键亮点

  • 使用启动锁避免同一 session 并发冷启动。
  • Fork 后立即 destroy wrapper，避免 SDK 内部状态被原地修改后污染旧 session。
  • 临时 turn_context 注入每轮引用文件、技能、模式，回合结束后恢复。
  • MCP runtime 按 cwd 缓存，引用计数释放。

核心风险

  • 访问 SDK 私有字段，如 _baseSystemPrompt、_toolRegistry，升级 SDK 时需要回归。
  • 文件体量大，职责过多，后续维护成本高。
  • 状态机依赖事件顺序，需持续保持 SSE 与 SDK 事件兼容。

05 Session 读取、缓存与 SSE 事件流

Session 列表缓存

30 秒 TTL + stale-while-revalidate，失效时后台刷新，避免 UI 频繁扫描会话目录。

Session 文件缓存

以 path + mtimeMs + size 作为缓存键，最多 64 个文件，避免大 jsonl 重复解析。

大图片优化

读取会话时剥离 base64 大图，仅保留占位，降低 HTTP 响应体积。

06 多 Agent 协作系统

parallel-agent 是项目中最有扩展潜力的子系统。它支持只读调研、隔离编码、多方案并行、审查任务，并把 coding 模式 worker 放入独立 git worktree，产出 diff 供主会话审阅应用。

能力模式

  • ask：只读调研。
  • code：隔离编码，单 worker。
  • parallel：方案 A/B/C 多 worker 独立实现。
  • review：审查、安全或 CI 调查。

持久化与恢复

任务写入 ~/.deerhux/agent/tasks/*.jsonl。应用关闭后仍可加载，运行中断会标记为 recoverable。

07 定时任务调度系统

调度器基于 node-cron，在 Next instrumentation 启动时注册。它用文件锁避免 Tauri/Next/HMR 下重复执行，任务数据写入 scheduled-tasks.json。

文件职责评价

08 MCP、CodeGraph 与工具体系

工具体系由内置编码工具、CodeSearch、CodeGraph、MCP runtime tools、spawn_subagent 组成。full preset 会动态包含 MCP 工具。

MCP Runtime

支持 stdio 传输，读取多个配置源：用户全局、DeerHux agent 目录、项目 .pi、项目 .deerhux。SSE/HTTP transport 已识别但尚未实现。

CodeGraph

若当前项目 CodeGraph 已初始化，会注册 status/search/callers/callees/impact 五个工具，并提示模型优先用于符号查询。

09 API 路由地图

项目共有 66 个 route.ts、96 个 HTTP 方法。API 按能力域组织，agent 与 sessions 是主链路，subagent/isolated/parallel 是协作链路。

10 前端组件层

前端是高度状态化的工作台 UI。AppShell 管理全局布局、标签、会话选中、项目 cwd；ChatWindow 管理会话展示与运行态；ChatInput 是复杂输入控制中心。

前端主要问题不是类型错误，而是文件过大。ChatInput、AppShell、SessionSidebar、MessageView 都已接近或超过 1300 行，建议后续按状态域拆 Hook、按 UI 区块拆子组件。

12 安全边界与风险点

已有安全边界

  • 文件访问通过 allowed roots 限制，来源包括会话 cwd、动态加入根、旧默认 cwd。
  • Scheduler 有进程锁和任务运行锁。
  • MCP 使用 spawn 且 shell:false。
  • 图片上传落盘并减少 session 内 base64 膨胀。

需要关注的风险

  • worktree.ts 部分 git 命令使用 execSync 字符串拼接，建议改 execFileSync 参数数组。
  • Tauri CSP 为 null，需确认不加载不可信远程内容。
  • MCP 服务 command/args 来自用户配置，属于高权限本地执行能力，应在 UI 明确提示。
  • rpc-manager 访问 SDK 私有字段，升级依赖可能破坏行为。
风险项位置等级建议
命令字符串拼接lib/parallel-agent/worktree.ts高全部改为 execFileSync("git", [args...])，避免路径转义问题。
组件超大components/*.tsx中拆分 hooks、presentational components、reducers。
SDK 私有字段依赖lib/rpc-manager.ts中封装 compat 层并补充单测。
JSON 文件并发写scheduler/store, roles, memory 等中关键配置写入可采用原子写 tmp + rename。

结论

DeerHux 当前已经形成了完整的本地 AI Agent 桌面工作台架构：前端 UI、Next API、AgentSession 生命周期、事件流、工具系统、多 Agent 编排、定时任务、MCP 扩展、桌面发布路径都已闭环。代码质量整体较高，类型检查通过且无明显未完成标记。后续主要应围绕安全硬化、核心模块拆分、测试覆盖与扩展协议完善继续演进。
