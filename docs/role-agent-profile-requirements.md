# 角色 / Agent Profile 功能需求文档

## 1. 背景

当前软件已有「角色」相关入口，但角色能力尚未形成清晰的用户路径。用户希望角色不只是一个可手动编辑的系统提示词模板，而是能够在日常对话中自然沉淀设定。

新的产品方向是：

> 用户在输入框下方选择当前角色；当用户在对话中表达“给这个角色增加设定 / 以后这个角色要这样做 / 记住这个角色设定”等意图时，AI 自动识别可沉淀的角色设定，并先向用户确认；用户确认后，该设定进入当前角色的「角色设定库」。之后其他 session 使用同一角色时，也能继承这些设定。

该功能的核心不是单纯“切换角色”，而是建立一个 **可选择、可确认、可沉淀、跨会话复用** 的角色设定系统。

本需求文档仅描述产品需求与交互逻辑，不涉及具体技术实现。

---

## 2. 产品目标

### 2.1 核心目标

在输入框下方提供角色选择能力，并支持 AI 从用户对话中自动识别角色设定，经用户确认后存入当前角色的角色设定库，使该角色在其他 session 中也能持续使用这些设定。

### 2.2 用户价值

- 用户可以在发消息前明确选择本次对话使用的角色。
- 默认情况下使用「默认角色」，降低理解成本。
- 用户不需要进入复杂设置页，也能通过自然语言给角色增加设定。
- AI 不会擅自写入角色设定，所有沉淀都需要用户确认。
- 角色设定可以跨 session 复用，减少重复说明。
- 用户可以逐步培养自己的专属角色。

### 2.3 产品定位

该功能定位为：

> 面向 AI Coding Agent 的角色选择与角色设定沉淀系统。

它同时包含两类能力：

1. **角色选择**：用户在输入框下方选择当前 session 使用的角色。
2. **角色设定库**：用户通过自然语言持续为角色补充可复用的系统提示词设定。

## 3. 核心用户路径

### 3.1 默认路径

用户进入任意 session 后，输入框下方默认展示当前角色选择器。

默认状态：

```text
角色：默认角色 ˅
```

用户可以直接输入消息并发送，此时系统使用默认角色。

### 3.2 切换角色路径

用户点击输入框下方的角色下拉选择器。

示意：

```text
角色：默认角色 ˅

下拉列表：
✓ 默认角色
  编程专家
  UI/UX 设计师
  产品经理
  代码审查员
  只读分析师

+ 创建新角色
管理角色设定
```

用户选择某个角色后，当前 session 后续对话使用该角色。

示例：

```text
角色：产品经理 ˅
```

### 3.3 通过自然语言创建或补充角色设定

当用户在消息中表达对角色的设定诉求时，AI 自动识别。

用户示例：

```text
以后产品经理这个角色回答时，先帮我拆用户价值，再说实现成本。
```

或：

```text
给当前角色加一个设定：回答需求文档时要包含边界情况和验收标准。
```

或：

```text
这个 UI/UX 设计师以后要多给几个方案，不要只给一个结论。
```

AI 识别后，不应立即写入，而是先向用户确认。

确认示例：

```text
我识别到你想为当前角色「产品经理」新增一条角色设定：

「回答产品需求时，先拆解用户价值，再说明实现成本。」

是否存入「产品经理」的角色设定库？

[确认存入] [仅本次对话使用] [取消]
```

用户点击或回复确认后，该设定才进入角色设定库。

### 3.4 跨 session 复用路径

当其他 session 选择同一个角色时，该角色的角色设定库自动参与角色系统提示词。

示例：

用户在 session A 中为「产品经理」存入设定：

```text
回答需求时先拆用户价值，再说实现成本。
```

用户在 session B 中选择「产品经理」后，AI 应自动继承该设定。

## 4. 设计原则

### 4.1 角色选择入口靠近输入框

角色选择应放在输入框下方或输入区域附近，因为角色决定“下一条消息由哪个角色响应”。

该入口比左下角固定按钮更贴近用户发送消息的动作路径。

### 4.2 默认角色永远可用

所有 session 默认使用「默认角色」。用户即使不了解角色功能，也可以正常聊天。

### 4.3 AI 可以识别设定，但不能擅自写入

AI 可以主动识别用户话语中的角色设定意图，但必须先确认。

未经用户确认，不得写入角色设定库。

### 4.4 角色设定应跨 session 生效

角色设定库属于角色，不属于单一 session。

当用户确认把设定存入某个角色后，其他 session 使用该角色时也应继承。

### 4.5 当前 session 与角色设定库要区分

用户可能只希望某个设定临时用于当前对话，不希望写入角色库。

因此确认时应提供：

- 确认存入角色设定库
- 仅本次对话使用
- 取消

### 4.6 对用户保持透明

用户应能查看当前角色有哪些设定、哪些设定会影响系统提示词。

## 5. 功能范围

### 5.1 第一版必须包含

1. 输入框下方展示角色选择器。
2. 默认选中「默认角色」。
3. 用户可通过下拉选择角色。
4. 用户可创建新角色。
5. 用户消息中包含角色设定意图时，AI 自动识别。
6. AI 将识别到的设定整理成可存储的设定文本。
7. AI 应判断该设定应该落入哪个角色设定分块，例如 Identity、Soul、Rules、User、Tools、Memory。
8. 写入角色设定库前必须向用户确认。
9. 用户确认后，设定进入当前角色的角色设定库对应分块。
10. 其他 session 使用同一角色时，可继承该角色设定库。
11. 用户可查看当前角色的角色设定库。
12. 用户可删除或编辑角色设定库中的设定。

### 5.2 第一版可选包含

1. 角色设定分类。
2. 角色设定启用 / 禁用开关。
3. 角色设定来源 session 展示。
4. 角色设定创建时间展示。
5. 角色设定冲突提醒。

### 5.3 暂不包含

第一版暂不建议包含：

- 角色市场
- 多用户共享角色
- 子 Agent 管理
- 任务队列
- 浏览器 Profile
- 复杂长期记忆系统
- Prompt 版本回滚
- 高级 Prompt 变量编排

## 6. 角色与角色设定库

### 6.1 角色是什么

角色是 AI 的行为配置单元。

一个角色至少包含：

- 角色名称
- 角色描述
- 基础系统提示词
- 角色设定库

示例：

```text
角色名称：产品经理
角色描述：擅长需求拆解、产品规划、用户路径和优先级判断
基础系统提示词：你是一个资深产品经理...
角色设定库：
- 回答需求时先拆用户价值，再说明实现成本。
- 输出 PRD 时必须包含边界情况和验收标准。
```

### 6.2 角色设定库是什么

角色设定库是某个角色的可复用设定集合。

它的特点：

- 由用户确认后写入。
- 可以跨 session 生效。
- 可以被查看、编辑、删除。
- 会参与该角色最终系统提示词。

### 6.3 角色设定库的分块结构

角色设定库必须采用分块结构，而不是简单的一堆无序文本。

分块的目的是：

- 让用户知道每条设定影响 AI 的哪个方面。
- 让 AI 在写入设定时可以自动归类。
- 让最终系统提示词可以按照稳定顺序拼接。
- 方便用户后续查看、编辑、删除某一类设定。

参考截图中的文档式结构，建议将角色设定库拆分为以下模块：

| 模块 | 文件感命名 | 含义 | 第一版必要性 |
|---|---|---|---|
| Identity | `IDENTITY.md` | Agent 身份、职责、专业领域、角色定位 | 必须 |
| Soul | `SOUL.md` | 性格、语气、表达风格、回答习惯 | 必须 |
| Rules | `RULES.md` | 行为规则、工作方式、输出约束 | 必须 |
| User | `USER.md` | 用户偏好、协作习惯、长期偏好 | 必须 |
| Tools | `TOOLS.md` | 工具使用原则、操作边界、执行习惯 | 建议必须 |
| Memory | `MEMORY.md` | 该角色专属长期记忆、跨 session 沉淀信息 | 可选但建议保留入口 |
| Agents / Workspace | `AGENTS.md` | 工作区规则、项目级 standing orders | 不属于角色库，但应参与最终拼接 |
| Heartbeat | `HEARTBEAT.md` | 维护提醒、周期性自检、轻量运行指导 | 第一版不必要 |

### 6.4 第一版必要分块

第一版建议不要完全照搬截图里的所有模块，而是保留真正会影响角色系统提示词的必要分块。

第一版必要：

1. **Identity / 身份与职责**
2. **Soul / 语气与风格**
3. **Rules / 行为规则**
4. **User / 用户偏好**
5. **Tools / 工具使用规则**

第一版可选：

1. **Memory / 角色长期记忆**

第一版不建议做为角色库模块：

1. **Agents / Workspace Rules**：更适合作为项目级规则，不应归属于某个角色。
2. **Heartbeat**：更适合作为后续 Agent 运行维护能力，第一版不必要。

### 6.5 各分块说明

#### 6.5.1 Identity / 身份与职责

用于定义角色是谁、擅长什么、负责什么。

适合写入的设定示例：

```text
这个角色是资深产品经理，擅长需求拆解、用户路径和优先级判断。
```

#### 6.5.2 Soul / 语气与风格

用于定义角色如何表达。

适合写入的设定示例：

```text
以后这个角色回答时先给结论，再解释原因，语气直接、专业。
```

#### 6.5.3 Rules / 行为规则

用于定义角色做事方式和输出规则。

适合写入的设定示例：

```text
输出 PRD 时必须包含背景、目标、范围、边界情况和验收标准。
```

#### 6.5.4 User / 用户偏好

用于记录用户希望该角色长期遵循的协作偏好。

适合写入的设定示例：

```text
用户偏好简洁回答，除非明确要求，不要输出过长解释。
```

#### 6.5.5 Tools / 工具使用规则

用于定义角色使用工具前后的行为边界。

适合写入的设定示例：

```text
修改代码前必须先说明修改方案，得到确认后再动手。
```

#### 6.5.6 Memory / 角色长期记忆

用于沉淀该角色专属、跨 session 有价值的信息。

适合写入的设定示例：

```text
该角色主要服务于 SaaS 产品需求文档和增长实验方案设计。
```

### 6.6 角色设定库不是什么

角色设定库不应等同于普通聊天记忆。

它不应该自动记录所有用户偏好，也不应该记录未经确认的临时内容。

只有满足以下条件的内容才应进入角色设定库：

1. 用户表达了明确或强烈的角色设定意图。
2. AI 提取出明确、可复用、适合角色的设定。
3. 用户确认存入。

### 6.7 写入分块规则

AI 识别到设定后，不仅要提取设定内容，还要判断它应写入哪个分块。

确认时应展示分块归属。

示例：

```text
我识别到你想为「产品经理」新增角色设定：

分块：Rules / 行为规则
设定：输出 PRD 时必须包含边界情况和验收标准。

是否存入「产品经理」角色设定库？

[确认存入] [改到其他分块] [仅本次对话使用] [取消]
```

如果分块判断不确定，应允许用户调整。

### 6.8 分块拼接顺序

角色设定库参与系统提示词时，应按固定顺序拼接，避免不同 session 中表现不一致。

推荐顺序：

```text
Identity
Soul
Rules
User
Tools
Memory
```

其中 Memory 可为空；空分块在最终提示词中可以不展示。

## 7. 角色设定识别规则

### 7.1 应识别的表达

AI 应识别以下类型的话语：

#### 7.1.1 明确设定类

```text
给这个角色加一个设定：以后回答要更简洁。
```

```text
把“输出需求时必须包含验收标准”存到产品经理角色里。
```

#### 7.1.2 习惯要求类

```text
以后这个角色都先给结论，再解释原因。
```

```text
以后 UI/UX 设计师回答时要多给几个方案。
```

#### 7.1.3 角色身份类

```text
这个角色要扮演一个严格的代码审查员。
```

```text
让这个角色更像一个有经验的产品负责人。
```

#### 7.1.4 输出格式类

```text
以后用这个角色写 PRD 时，都要包含背景、目标、范围、验收标准。
```

#### 7.1.5 工具行为类

```text
这个角色以后修改代码前必须先给我方案。
```

### 7.2 不应识别为角色设定的表达

以下内容不应自动触发角色设定确认：

- 一次性任务要求
- 当前问题的上下文说明
- 普通聊天内容
- 项目临时信息
- 明显只针对当前回答的格式要求

示例：

```text
这次回答简洁一点。
```

不应默认存入角色设定库，除非用户明确表示“以后都这样”。

### 7.3 需要澄清的表达

如果用户表达模糊，AI 应先澄清，不应直接确认写入。

示例：

用户：

```text
这个角色以后专业一点。
```

AI：

```text
你希望「专业一点」具体体现在哪些方面？例如：
1. 更少口语化表达
2. 多给依据和风险说明
3. 使用更正式的文档结构
4. 更严格指出问题
```

## 8. 确认机制

### 8.1 标准确认卡片

当 AI 识别到角色设定意图时，应展示确认卡片。

示意：

```text
识别到角色设定

当前角色：产品经理

建议存入的设定：
「回答产品需求时，先拆解用户价值，再说明实现成本。」

建议分块：Rules / 行为规则

该设定将影响之后所有使用「产品经理」角色的 session。

[确认存入] [改到其他分块] [仅本次对话使用] [取消]
```

### 8.2 确认存入

用户选择「确认存入」后：

- 设定进入当前角色的角色设定库。
- 当前 session 后续回复也使用该设定。
- 其他 session 使用该角色时也使用该设定。

反馈文案：

```text
已存入「产品经理」角色设定库。之后所有使用该角色的会话都会应用该设定。
```

### 8.3 仅本次对话使用

用户选择「仅本次对话使用」后：

- 设定只影响当前 session。
- 不写入角色设定库。
- 其他 session 不受影响。

反馈文案：

```text
已仅应用于当前对话，不会写入角色设定库。
```

### 8.4 取消

用户选择「取消」后：

- 不写入角色设定库。
- 不改变当前角色设定。

反馈文案：

```text
已取消，不会保存该角色设定。
```

### 8.5 多条设定确认

如果用户一次提出多条设定，AI 应拆分成多条，并允许用户分别确认。

示例：

```text
识别到 3 条角色设定：

[✓] 回答前先给结论。
[✓] 修改代码前先说明方案。
[ ] 输出内容尽量使用表格。

[确认选中设定] [仅本次对话使用] [取消]
```

## 9. 输入框区域设计

### 9.1 基础布局

角色选择器位于输入框下方，靠近发送操作区域。

示意：

```text
┌──────────────────────────────────────┐
│ 输入你的问题...                         │
└──────────────────────────────────────┘
角色：默认角色 ˅       工具：默认      模型：Claude
```

如果当前已有工具、模型、thinking 等控制项，角色应与这些控制项处于同一层级。

### 9.2 角色选择器状态

#### 默认状态

```text
角色：默认角色 ˅
```

#### 已选择角色

```text
角色：产品经理 ˅
```

#### 当前角色有新增未确认设定

```text
角色：产品经理 ˅   有 1 条设定待确认
```

#### 当前角色设定库为空

```text
角色：产品经理 ˅   暂无角色设定
```

### 9.3 下拉菜单内容

角色下拉菜单应包含：

```text
选择角色

✓ 默认角色
  编程专家
  UI/UX 设计师
  产品经理
  代码审查员
  只读分析师

────────────
+ 创建新角色
查看当前角色设定库
管理全部角色
```

### 9.4 当前角色设定库入口

用户可以从下拉菜单进入当前角色设定库。

示意：

```text
产品经理设定库

Identity / 身份与职责
- 你是一个资深产品经理，擅长需求拆解、产品规划、用户路径和优先级判断。

Soul / 语气与风格
- 回答时先给结论，再解释原因。

Rules / 行为规则
- 回答需求时先拆用户价值，再说明实现成本。
- 输出 PRD 时必须包含边界情况和验收标准。

User / 用户偏好
- 用户偏好简洁、直接、可执行的建议。

Tools / 工具使用
- 修改代码前先说明方案。

Memory / 长期记忆，可选
- 暂无

[新增设定] [编辑] [删除]
```

## 10. 创建角色路径

### 10.1 从下拉菜单创建

用户点击角色下拉菜单中的「创建新角色」。

创建弹窗示意：

```text
创建角色

角色名称
[ 例如：增长产品经理 ]

角色描述
[ 擅长增长策略、数据分析和实验设计 ]

基础设定，可选
[ 你是一个资深增长产品经理... ]

[取消] [创建并使用]
```

### 10.2 创建后默认选中

用户创建新角色后，该角色应自动成为当前 session 选中的角色。

反馈：

```text
已创建并切换到「增长产品经理」。
```

### 10.3 通过对话补充新角色设定

创建角色后，用户可以继续通过自然语言补充。

示例：

```text
以后这个角色做增长方案时，都要给出指标、实验假设和验证方法。
```

AI 应识别并确认是否存入该新角色设定库。

## 11. 最终系统提示词组成

当某个 session 使用某个角色时，最终影响 AI 的角色部分必须由「分块设定」按稳定顺序拼接，而不是把所有设定无序混在一起。

### 11.1 角色部分拼接结构

角色部分建议由以下内容组成：

```text
角色基础设定
+ Identity / 身份与职责
+ Soul / 语气与风格
+ Rules / 行为规则
+ User / 用户偏好
+ Tools / 工具使用规则
+ Memory / 角色长期记忆，可选
+ 当前 session 临时设定
```

示意：

```text
# Role: 产品经理
你是一个资深产品经理，擅长需求拆解、产品规划、用户路径和优先级判断。

# Identity
- 你是资深产品经理，关注用户价值、实现成本和迭代路径。

# Soul
- 回答时先给结论，再解释原因。
- 语气直接、专业、克制。

# Rules
- 回答需求时先拆用户价值，再说明实现成本。
- 输出 PRD 时必须包含边界情况和验收标准。

# User
- 用户偏好简洁、清晰、可执行的建议。

# Tools
- 修改代码前先说明方案。

# Memory
- 该角色主要服务于 SaaS 产品需求文档和增长实验方案设计。

# Session Temporary Settings
- 本次回答请更偏向商业化分析。
```

### 11.2 全局最终 Prompt 来源

完整系统提示词应区分「项目级规则」和「角色级设定」。

推荐来源顺序：

1. 基础系统规则
2. 项目级规则 / Workspace Rules / `AGENTS.md`
3. 当前角色基础设定
4. 当前角色设定库：Identity
5. 当前角色设定库：Soul
6. 当前角色设定库：Rules
7. 当前角色设定库：User
8. 当前角色设定库：Tools
9. 当前角色设定库：Memory，可选
10. 当前 session 临时设定
11. 用户当前消息

### 11.3 必须保证分块可预览

用户查看最终系统提示词时，应能看到每个分块的来源和内容。

示意：

```text
最终系统提示词预览

来源：
- Workspace Rules / AGENTS.md
- Role: 产品经理
- Identity
- Soul
- Rules
- User
- Tools
- Memory
- Session Temporary Settings

--------------------------------

# Workspace Rules
...

# Role Identity
...

# Role Soul
...
```

### 11.4 设定冲突

如果新设定与已有设定冲突，AI 应提示用户选择处理方式。

示例：

```text
检测到新设定可能与已有设定冲突：

已有设定：回答保持简洁，只给关键结论。
新设定：回答时尽量详细，展开说明背景和原因。

请选择：
[替换旧设定] [保留两者] [仅本次使用] [取消]
```

## 14. 边界情况

### 14.1 用户未选择角色

默认使用「默认角色」。

### 14.2 当前消息同时包含任务和设定

例如：

```text
以后这个角色回答 PRD 都要带验收标准。现在帮我写一个登录页 PRD。
```

AI 应先识别设定并确认，同时可以继续完成当前任务。

推荐处理：

1. 对当前任务临时应用该设定。
2. 同时询问是否存入角色设定库。

### 14.3 用户提到其他角色

如果当前选中角色是「默认角色」，但用户说：

```text
给产品经理角色加一个设定：以后先说用户价值。
```

AI 应识别目标角色是「产品经理」，确认时明确目标角色。

```text
是否将该设定存入「产品经理」角色设定库？
```

### 14.4 目标角色不存在

如果用户提到的角色不存在，应询问是否创建。

```text
未找到「增长产品经理」角色。是否创建该角色并存入这条设定？

[创建并存入] [仅本次使用] [取消]
```

### 14.5 设定过长或包含多件事

AI 应拆分并归纳为清晰设定，供用户确认。

### 14.6 用户撤销设定

用户应能在角色设定库中删除设定。

如果用户在聊天中说：

```text
把产品经理角色里“先说实现成本”那条删掉。
```

AI 应先确认删除，再执行。

### 14.7 当前正在生成回复

如果 AI 正在回复中切换角色，应提示：

```text
当前正在生成回复。角色切换将在下一次用户消息后生效。
```

## 15. 视觉与体验要求

### 15.1 风格方向

整体保持轻量、清晰、低干扰。

角色选择器不应喧宾夺主，应像模型选择、工具选择一样成为输入区域的辅助控制项。

### 15.2 角色选择器

建议样式：

```text
角色：默认角色 ˅
```

或更紧凑：

```text
@ 默认角色 ˅
```

### 15.3 设定确认卡片

确认卡片应突出：

- 当前目标角色
- 建议存入的设定
- 生效范围
- 用户可选操作

### 15.4 角色设定库

角色设定库应以列表形式呈现，每条设定短句化、可编辑。

示意：

```text
产品经理设定库

1. 回答需求时先拆用户价值，再说明实现成本。
2. 输出 PRD 时必须包含边界情况和验收标准。
3. 涉及优先级时，说明推荐排序依据。
```

## 16. 成功标准

第一版完成后，应满足：

1. 用户可以在输入框下方看到当前角色。
2. 默认角色默认选中。
3. 用户可以通过下拉菜单切换角色。
4. 用户可以创建新角色。
5. 用户可以用自然语言表达角色设定。
6. AI 可以识别角色设定意图并提取设定内容。
7. AI 不会未经确认写入角色设定库。
8. 用户确认后，设定可以存入当前角色设定库。
9. 同一角色在其他 session 中可以继承已保存设定。
10. 用户可以查看、编辑、删除角色设定。
11. 角色设定库按 Identity、Soul、Rules、User、Tools、Memory 等分块管理。
12. 最终系统提示词按固定分块顺序拼接。
13. 临时设定和角色库设定有明确区别。
14. 整体流程不干扰用户正常发送消息。

---

## 17. 推荐第一版方案总结

推荐第一版采用：

> 输入框下方角色选择器 + 默认角色 + 角色下拉切换 + 自然语言设定识别 + 用户确认写入 + 角色设定库跨 session 复用

第一版的关键不是做一个复杂的 Agent 管理后台，而是让用户能够在最自然的聊天路径里完成角色培养：

1. 先选择角色。
2. 在对话中说出希望角色以后怎么做。
3. AI 自动识别可沉淀设定。
4. 用户确认。
5. 设定进入角色设定库。
6. 其他 session 继续使用同一角色时自动继承。

这条路径足够轻量，也能真正实现“角色直接控制系统提示词”。
