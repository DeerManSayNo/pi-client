② 读写流转变迁

整改前 · 单一瓶颈

Agent 写、Subagent 写、UI 读、polling 读 全部竞争同一批 JSONL，冷启动 / cache miss 必须全量扫描 → 30s 超时、打开慢。

sidebar ──► listAll() └─► 全量扫描 JSONL 打开session ──► GET /:id?includeState └─► 全量解析 + 阻塞等 runtime

整改后 · 分层解耦

JSONL 只在 rebuild 时回源；左侧栏读 index；打开 session = 分页消息 + 异步 state；runtime 永不阻塞首屏。

sidebar ──► 读 index（O(1)） └─► stale→后台 rebuild（不阻塞） 打开session ─► GET /:id/messages（最近100） ├─► 首屏立刻显示 └─► GET /:id/state（异步，5s超时）

⑥ 可回滚设计（三层 feature flag）

DEERHUX_SESSION_INDEX

=0 时 /api/sessions 回退 listAllSessions() 全量扫描旧逻辑。

DEERHUX_SESSION_PAGING

=0 时 messages 接口返回全部消息，首屏不分页。

DEERHUX_SESSION_TRACE

=0 时 trace 全部 no-op（默认关闭）。
